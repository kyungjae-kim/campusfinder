MSA-React 미니 프로젝트 과제: 캠퍼스 분실물 중개 플랫폼
0. 과제 목표

캠퍼스에서 발생하는 분실물/습득물을 웹으로 등록하고, 매칭 후보를 확인한 뒤, 안전하게 인계(수령)까지 진행할 수 있는 서비스를 만든다.
백엔드는 서비스 9개로 분리하고, 프론트는 React로 화면을 많이 구성한다.
트랜잭션/분산 트랜잭션은 다루지 않는다. 핵심은 역할(권한) + 상태(워크플로우) + 화면 다양성이다.

1. 필수 사용자 그룹(6종)

아래 6개 역할을 반드시 구현한다.

LOSER(분실자): 내 분실 신고 등록, 매칭 확인, 인계 요청

FINDER(습득자): 내 습득물 등록, 인계 승인/거절

OFFICE(관리실 직원): 보관/접수 처리, 인계 승인, 민원 대응

SECURITY(보안/경비): 위험물/고가품 확인, 인계/보관 승인(검수 역할)

ADMIN(시스템 관리자): 신고/게시물 관리, 사용자 제재, 운영 통계

COURIER(외부 인계 파트너, 옵션이지만 역할은 필수): 대면 인계가 어려울 때 “대리 인계(택배/보관함)” 진행 상태만 처리

COURIER는 실제 택배 연동 금지. “상태 변경 + 조회”만 담당한다.

2. 권한 규칙(필수)
2-1. 연락처/개인정보 공개 원칙

분실자/습득자는 서로의 연락처(전화/이메일)를 기본적으로 볼 수 없다.

아래 조건을 만족할 때만 연락처가 공개된다.

인계 요청이 생성되고

습득자(FINDER)가 승인하고

(정책상 필요한 경우) 관리실(OFFICE) 또는 보안(SECURITY)이 최종 승인했을 때

3. 백엔드 서비스 구성(9개, 필수)

Auth 서비스: 로그인/토큰 발급, 역할(Role) 포함

User 서비스: 사용자 프로필, 역할, 상태(활성/정지), 소속(학생/교직원)

Lost 서비스: 분실 신고 CRUD, 상태 변경

Found 서비스: 습득물 CRUD, 보관 정보 포함, 상태 변경

Matching 서비스: 분실↔습득 매칭 후보 산출(규칙 기반)

Handover 서비스: 인계 요청/승인/일정/완료 처리(상태 흐름의 중심)

Message 서비스: 인계 건별 채팅(간단한 메시지)

Notification 서비스: 이벤트 발생 시 알림 저장/조회/읽음 처리

Admin 서비스: 신고 처리, 블라인드, 통계, 사용자 제재

4. 핵심 엔티티와 상태(필수)
4-1. 분실 신고(Lost)

필드(최소): lostId, userId, category, title, description, lostAt(분실 시각), lostPlace(분실 장소), reward(사례금 여부/금액 옵션), status, createdAt

status: OPEN / MATCHED / CLOSED

4-2. 습득물(Found)

필드(최소): foundId, ownerUserId(습득자), category, title, description, foundAt, foundPlace, storageType(SELF/OFFICE/SECURITY/LOCKER), storageLocation, status, createdAt

status: REGISTERED / STORED / IN_HANDOVER / HANDED_OVER / DISCARDED

4-3. 인계(Handover)

필드(최소): handoverId, lostId, foundId, requesterId(분실자), responderId(습득자), method(MEET/OFFICE/COURIER), scheduleAt, meetPlace, status, approvedAt들

status:
REQUESTED → ACCEPTED_BY_FINDER → (OPTION) VERIFIED_BY_SECURITY → APPROVED_BY_OFFICE → SCHEDULED → COMPLETED
취소: CANCELED (어느 단계에서든 가능)

4-4. 신고/모더레이션(Report)

필드(최소): reportId, targetType(LOST/FOUND/MESSAGE), targetId, reporterId, reason, status(OPEN/RESOLVED), createdAt

5. 매칭 규칙(쉬운 버전, 필수)

Matching 서비스는 “예측” 금지. 아래 규칙 기반으로 후보를 만든다.

같은 category는 가산점

장소 문자열이 같은 건물/구역 키워드를 포함하면 가산점

날짜가 7일 이내로 가까우면 가산점

title/description 키워드가 1개 이상 겹치면 가산점

결과는 점수 높은 순으로 TOP N(예: 10개)을 반환한다.
정확도는 평가하지 않고, “동작”과 “정렬”만 평가한다.

7. 기능 요구사항(필수)
A. 인증/권한

A1. 로그인 시 토큰을 발급한다.
A2. 모든 API는 역할 기반 접근 제어를 적용한다.
A3. 정지된 사용자(status=BLOCKED)는 글 등록/메시지/인계 요청이 불가능하다.

B. 분실 신고

B1. 분실자는 분실 신고를 등록/수정/삭제(논리삭제 가능)할 수 있다.
B2. 분실 신고는 기본 상태 OPEN이며, 인계 완료 시 CLOSED로 전환된다.
B3. 목록은 페이지네이션을 제공한다.

C. 습득물

C1. 습득자는 습득물을 등록/수정/삭제할 수 있다.
C2. 관리실은 “대리 등록”이 가능하다(분실물 센터에 직접 맡긴 케이스).
C3. 습득물은 보관 방식(storageType)과 보관 위치(storageLocation)를 가진다.

D. 매칭

D1. 분실 신고 상세에서 매칭 후보를 볼 수 있다.
D2. 매칭 후보에서 “인계 요청”을 생성할 수 있다.
D3. OFFICE/SECURITY는 “수동 매칭 연결”(lostId, foundId 연결) 기능을 가진다(간단 버튼).

E. 인계(핵심)

E1. 분실자는 후보 습득물에 대해 인계 요청을 생성한다(REQUESTED).
E2. 습득자는 요청을 승인/거절한다.
E3. 인계 방법:

MEET: 일정/장소를 정하고 대면 인계

OFFICE: 관리실에서 인계(OFFICE 승인 필요)

COURIER: 외부 파트너가 단계 상태만 처리

E4. SECURITY 검수 규칙(쉬운 버전)

category가 “전자기기/지갑/신분증”이면 SECURITY 검수 단계가 필수(VERIFIED_BY_SECURITY)

그 외는 SECURITY 단계 없이 OFFICE 승인으로 진행 가능

E5. 인계 완료 시:

Handover는 COMPLETED

Lost는 CLOSED

Found는 HANDED_OVER

F. 메시지(채팅)

F1. 채팅은 “인계 건(handoverId)”이 생성된 뒤에만 가능하다.
F2. 채팅 참여자는 해당 인계의 당사자(LOSER/FINDER) + OFFICE/SECURITY(필요 시)만 가능하다.
F3. 욕설/도배 탐지 금지. 대신 신고 기능으로 처리한다.

G. 알림

G1. 아래 이벤트 발생 시 Notification에 저장한다.

분실 신고 등록 완료

습득물 등록 완료

매칭 후보 생성(선택: 매일 1회 정도로만)

인계 요청 생성/승인/거절/일정 확정/완료

SECURITY 검수 결과

ADMIN 블라인드/정지 조치

G2. 알림은 읽음 처리 가능해야 한다.

H. 신고/운영(ADMIN)

H1. 사용자는 게시물/메시지를 신고할 수 있다.
H2. ADMIN은 신고를 처리하고, 대상 글/메시지를 블라인드할 수 있다.
H3. ADMIN은 사용자 정지/해제를 할 수 있다.
H4. 운영 통계 화면에서 최소 3개 지표 제공:

기간별 분실 신고 수

기간별 습득물 등록 수

인계 완료 수

8. 제출 필수 테스트 시나리오(통과해야 함)

분실자가 분실 신고 등록 → 목록/상세에서 조회됨

습득자가 습득물 등록 → 목록/상세에서 조회됨

분실 신고 상세에서 매칭 후보가 점수순으로 보임

분실자가 인계 요청 생성 → 습득자 수신함에 표시됨

습득자가 승인 → 인계 상태가 다음 단계로 진행됨

전자기기/지갑/신분증 카테고리는 SECURITY 검수 단계가 끼어야 함

승인 완료 전에는 연락처가 마스킹되어야 함, 승인 완료 후에는 공개됨

인계 완료 처리 시 Lost=CLOSED, Found=HANDED_OVER로 동기화됨

정지된 사용자는 글 등록/인계 요청/메시지 전송이 막혀야 함

신고 등록 → ADMIN 신고 관리 화면에서 처리/블라인드 가능해야 함

이벤트 발생 시 알림함에 기록이 남아야 함(예약 발송/푸시 금지)

9. 가산점(선택)

즐겨찾기(관심 카테고리/장소) 기반 매칭 후보 우선 노출

사진 업로드(저장만, 이미지 분석 금지)

OFFICE용 “라벨 출력 흉내”(라벨 텍스트 생성)

COURIER 상태 흐름(픽업/이동/전달 완료) 완성

다국어(한국어/영어) 토글
